\section{Introduction}\label{sec:methodology_introduction}

This chapter explains the methodology adopted for planning, developing,
validating, and refining Eduverse as a graduation project. It focuses on the
practical development approach and workflow structure rather than on detailed
implementation artifacts, which are discussed later in the thesis.

Eduverse required repeated refinement because it combines the web application
as the primary client with an MVP mobile companion, backend services,
database migrations, role-based workflows, live sessions, AI-assisted
features, and an IDE-based coding workspace.

\section{System Development Methodology}\label{sec:system_development_methodology}

Eduverse is best described as having been developed through a hybrid
iterative workflow with predominantly feature-based vertical slices. This
description is more accurate than a strictly phase-based model because the
repository evidence shows repeated feature increments rather than a simple
sequence such as frontend first, backend later, and testing only at the end.
After an initial prototype and project setup stage, the system progressed
through iterative increments in which the selected module was implemented,
reviewed, and refined in relation to the wider platform.

The methodology remained incremental, but it was not organized as isolated
technical layers. When a feature required server-side behavior, permission
updates, or data-model changes, those concerns were usually developed close
to the same increment rather than postponed to a later project phase. Within
this process, the web application remained the primary execution environment,
while the mobile application was kept as an MVP companion for selected
high-frequency academic activities.

\subsection{Feature-Based Iterative Workflow}\label{subsec:feature_based_iterative_workflow}

Development generally proceeded through working slices organized around real
academic or administrative workflows. Examples include organization
management, class workflows, materials, assignments, examinations, live
sessions, IDE support, and AI-assisted features. In many increments, the
selected slice combined interface work with the corresponding API or server
logic, shared domain services, and Supabase migrations when required.

This pattern allowed the project to evolve through usable increments rather
than through disconnected partial implementations. At the same time, the
repository evidence also shows refactoring, infrastructure work, and later
refinements, so the methodology is most accurately described as
predominantly feature-based rather than exclusively vertical-slice in every
commit.

\subsection{Justification for Methodology Selection}\label{subsec:justification_for_methodology_selection}

This methodology suited Eduverse because the platform combines a
multi-organization structure, role-based access, and diverse academic
workflows with recurring integration between interface behavior, backend
logic, and data constraints. A rigid linear model would have made it harder
to adjust permissions, workflow boundaries, and feature integration as
requirements became clearer through implementation.

It also supported continuous validation. As features evolved, the project
needed repeated confirmation that interface behavior, backend logic,
permissions, schema changes, and shared services remained coherent. This was
especially important because some increments affected multiple workflows at
the same time.

\section{Development Workflow Structure}\label{sec:development_workflow_structure}

In practical terms, the workflow followed a recurring structure in which the
team selected a feature or workflow, implemented the required slice,
validated it, and integrated it into the main project history.

Repository evidence also shows that build, type-check, and format-check
validation was configured through GitHub Actions for pull requests and
updates to the main branch. In practice, validation also included targeted
\texttt{bun test} usage for selected higher-risk logic where such tests were
available, together with Biome-backed formatting support. The repository does
not support a claim of universal automated test coverage. Detailed CI/CD
behavior, deployment setup, and broader engineering artifacts are therefore
left for Chapter 6. Accordingly, this chapter describes the methodology as a
controlled workflow for iterative feature integration rather than as a set of
isolated development phases.
